AI & Learning Design

Designing Training for AI Agent Annotation

How I built annotation guidelines and a learning experience to help 50+ evaluators accurately assess AI agent performance across 1M+ data points.

Role

Content Author & Learning Designer

Timeline

2024 – Present

Audience

Data Annotators & Quality Auditors

Frameworks

Bloom's Taxonomy, Gagné's Nine Events, Kirkpatrick Model, Adult Learning Theory

Tools

Articulate Storyline, Articulate Rise, Adobe Suite, Confluence

The Challenge

An AI research team was developing an autonomous web agent capable of navigating websites, extracting information, and completing tasks on behalf of users. To improve the model, the team needed human evaluators to accurately annotate thousands of real-world agent workflows — assessing whether the AI successfully completed each task, identifying where it failed, and categorizing the type and severity of errors.

The problem: the evaluation process was complex. Annotators needed to distinguish between multiple workflow statuses (success, failure, unknown, out of scope, demo, guardrail), identify specific failure steps, classify errors across a severity hierarchy, and assign granular failure buckets — all while handling sensitive customer data under strict security protocols.

Key Challenges

  • Complex multi-step evaluation process with 7+ classification dimensions
  • 12+ failure bucket categories requiring nuanced judgment
  • Strict security requirements for handling customer data
  • Frequent guideline updates as the AI model evolved
  • Mixed audience: new hires needing onboarding + experienced annotators needing retraining on updates

My Approach

01

Needs Analysis & Content Audit

I started by mapping the full annotation workflow end-to-end, identifying every decision point an annotator faces. Using Bloom's Taxonomy, I analyzed the cognitive demands at each step — from basic recall (identifying workflow statuses) to evaluation-level judgment (classifying failure severity). I catalogued the types of errors being made and traced them back to gaps in the existing documentation.

02

Structured Guideline Design

I authored comprehensive annotation conventions organized around the annotator's actual workflow — not the system's architecture. Drawing on Gagné's Nine Events of Instruction, each section opens with context (gaining attention), states the objective, provides guided examples, and closes with practice scenarios. This structure supports adult learners who need to understand the "why" before the "how."

I built in a severity-based error taxonomy with explicit rules: success runs can only pair with low-impact or no-failure categories, while failure runs require high-impact classifications. Decision tables replaced dense paragraphs, making correct pairings scannable at a glance.

03

Edge Case Library

Real-world AI behavior is messy. I developed a library of edge cases covering scenarios like: workflows with minor errors but successful outcomes, ambiguous prompts, cookie popup handling, human-in-the-loop activations, and model hallucinations. Each edge case included the scenario, the correct evaluation, and the reasoning — turning abstract rules into concrete judgment calls.

04

Audit Framework & Quality Loop

I designed a structured audit strategy with a 7-question pass/fail rubric, standardized error codes with severity levels (Critical P0, Major P1, Minor P2), and clear scoring thresholds. Applying the Kirkpatrick Model, I evaluated training effectiveness across multiple levels — from learner reaction and knowledge retention to on-the-job behavior changes and business impact on annotation quality metrics.

05

Living Documentation & Change Management

AI models evolve fast, and guidelines need to keep pace. I implemented a release notes system with dated entries, clear change descriptions, and "UPDATED" / "ADDED" tags inline so annotators could quickly identify what changed. This reduced confusion during transitions and gave experienced annotators a fast path to the latest updates.

Design Decisions

Workflow-Aligned Structure

Rather than organizing by concept, I structured the guidelines to mirror the annotator's actual task flow — step 1 through step 9. This reduced cognitive load and made the doc usable as a real-time reference during annotation.

Decision Tables Over Paragraphs

For complex classification rules (like status + failure category combinations), I used scenario tables with columns for status, category, and reasoning. This made correct pairings scannable at a glance instead of buried in prose.

Model Capability Reference

I created a "what the model can and can't do" section — organized into CAN, CANNOT, SHOULDN'T, FORBIDDEN, and NOT YET GOOD AT. This gave annotators the context they needed to make fair evaluations without over- or under-penalizing the AI.

Inline Change Markers

Instead of expecting annotators to read full release notes, I added "UPDATED" and "ADDED" tags directly in the relevant sections. Experienced annotators could scan for changes without re-reading the entire document.

Key Takeaways

  • Complex evaluation tasks need structure that mirrors the evaluator's workflow, not the system's architecture
  • Edge cases are where most annotation errors happen — investing in a concrete edge case library pays off quickly
  • Living documentation with inline change markers is essential when guidelines evolve frequently
  • A structured audit framework creates a feedback loop that continuously improves both the guidelines and annotator performance

Reflection

This project reinforced something I believe deeply about learning design: the best training materials don't just tell people what to do — they help people make better judgments. Annotation work is fundamentally about decision-making under ambiguity, and the learning experience needed to reflect that. Adult learning theory was central here — these evaluators are experienced professionals who need autonomy, relevance, and immediate applicability.

If I were to iterate further, I'd build interactive scenario-based assessments in Articulate Storyline where annotators practice classifying real (anonymized) workflows before going live — with branching scenarios that mirror the decision complexity they face on the job. The edge case library is a strong foundation for that kind of practice-based learning.
